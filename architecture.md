# VibeOs — Архитектура

> AI-операционная система: рабочий стол в браузере в стиле macOS, где **каждое
> приложение — это HTML, который генерируется агентом на лету**. Окна — iframe'ы,
> которые можно двигать, ресайзить, сворачивать и закрывать. Нажатия внутри окон
> превращаются в события → промпт → новый HTML. Агент — локальный **Claude Code**.

---

## 1. Концепция

Классическая ОС запускает заранее написанные программы. VibeOs — нет. Когда
пользователь "запускает Notepad", не существует кода Notepad'а: shell просит агента
*галлюцинировать* интерфейс заметочника, получает HTML-фрагмент и рисует его в окне.
Когда пользователь жмёт кнопку внутри этого окна, событие уходит агенту, тот
"продолжает" приложение и присылает обновлённый HTML.

Три кита:

1. **Shell (рабочий стол)** — стабильный, написанный руками нативный код: меню-бар,
   dock, оконный менеджер, рамки окон (traffic lights). Это "ядро", оно НЕ
   генерируется.
2. **Apps (содержимое окон)** — эфемерный AI-сгенерированный HTML внутри iframe.
   Использует общий **Design System**, чтобы всё выглядело как единая ОС.
3. **Agent Bridge** — локальный backend, который превращает намерения пользователя
   в промпты, гоняет их через Claude Code и возвращает HTML.

---

## 2. Высокоуровневая схема

```
┌──────────────────────────── Browser ────────────────────────────┐
│  SHELL (нативный TS, НЕ генерируется)                             │
│  ┌────────────┐  menu bar · dock · wallpaper · window manager    │
│  │ WindowMgr  │                                                  │
│  └────────────┘                                                  │
│   ┌─────────────── Window (рамка рисуется shell'ом) ──────────┐  │
│   │ ● ● ●  Notepad                                            │  │
│   │ ┌──────────────────────────────────────────────────────┐ │  │
│   │ │  <iframe sandbox srcdoc=...>   ← AI-сгенерированный   │ │  │
│   │ │     + design-system.css                               │ │  │
│   │ │     + bridge.js (перехват кликов → postMessage)       │ │  │
│   │ └──────────────────────────────────────────────────────┘ │  │
│   └───────────────────────────────────────────────────────────┘ │
└───────────────────────────────┬──────────────────────────────────┘
            postMessage          │  WebSocket / HTTP (JSON)
        (iframe ↔ shell)         │  (shell ↔ backend)
                                 ▼
┌──────────────────────── Agent Bridge (Node) ─────────────────────┐
│  Intent Router → Prompt Builder → Session Store → Cache          │
│                          │                                        │
│                          ▼                                        │
│              Claude Code (Agent SDK / CLI headless)               │
│         системный промпт = правила VibeOs + Design System         │
└──────────────────────────────────────────────────────────────────┘
```

## 3. Компоненты

### 3.1 Shell / Desktop Environment
Нативный код (не генерируется AI). Отвечает за:
- **Menu bar** (верхняя панель):  логотип, имя активного приложения, часы, статус
  агента (idle / thinking / streaming).
- **Dock** — закреплённые "приложения" (Notepad, Finder, Calculator,
  Browser, Terminal, Settings…). Клик = `launch`-интент.
- **Spotlight (Command+Space)** — поиск/лаунчер: можно ввести **любое** название
  приложения, и агент сгенерирует его на лету (см. 3.8).
- **Launchpad** — сетка всех приложений, включая ранее сгенерированные из кэша
  (см. 3.9).
- **Wallpaper / рабочий стол**, иконки на десктопе.
- **Window Manager** (см. ниже).
- **Глобальный статус / лоадеры** — пока агент думает, окно показывает skeleton.

> **App Registry динамический.** Список приложений = встроенные (dock) +
> пользовательские, сгенерированные через Spotlight и сохранённые в **App Cache**
> (см. 3.10). Dock, Spotlight и Launchpad читают из одного реестра.

### 3.2 Window Manager
Сердце "оконности". Чистый TS, без iframe-зависимостей.
- Жизненный цикл окна: `create / focus / minimize / maximize(restore) / close`.
- **Drag** за титлбар, **resize** за 8 хэндлов/края.
- **Z-index / фокус**: клик поднимает окно; активное окно подсвечивает titlebar.
- **Traffic lights** (● красный=close, ● жёлтый=minimize, ● зелёный=maximize).
- Анимации genie/minimize в dock (по желанию, этап polish).
- Состояние окон в едином сторе (`WindowState[]`): id, title, appId, geometry
  (x,y,w,h), zIndex, state(normal/min/max), sessionId, текущий HTML, статус
  (loading/ready).
- Snapshot/restore геометрии для maximize/restore.

### 3.3 Iframe Apps + Bridge shim
- Контент окна = `<iframe>` с **`srcdoc`** (или blob URL), наполненный
  AI-сгенерированным HTML.
- В каждый iframe **shell инъектит обёртку**:
  ```
  <head> <link design-system.css> </head>
  <body data-window-id="...">
     <!-- AI HTML здесь -->
     <script> bridge.js </script>
  </body>
  ```
- **bridge.js** (статичный, пишем руками) — тонкий слой внутри iframe:
  - перехватывает `click` по элементам с `data-action` / submit форм / изменения
    инпутов;
  - сериализует событие + релевантный DOM-стейт (значения полей);
  - шлёт `parent.postMessage({type:'vibe-event', ...}, '*')`;
  - принимает команды от shell (например, "покажи спиннер на кнопке").
- **Изоляция**: `sandbox="allow-scripts"` (без `allow-same-origin`) → iframe в
  opaque-origin, не имеет доступа к cookie/DOM родителя, но `postMessage` работает.

### 3.4 Design System (ключевой для консистентности)
Чтобы галлюцинированный HTML выглядел как единая macOS-подобная ОС, агент НЕ рисует
произвольный CSS — он **собирает UI из готовых классов**.
- `design-system.css` — переменные (цвета, тени, радиусы, типографика SF Pro),
  компоненты: `.vibe-btn`, `.vibe-toolbar`, `.vibe-list`, `.vibe-input`,
  `.vibe-card`, `.vibe-sidebar`, `.vibe-table`, `.vibe-tabs`, `.vibe-segmented`,
  `.vibe-menu`, иконки SF-symbols-like (SVG-спрайт или icon-font).
- Контракт элементов событий: интерактивные элементы помечаются
  `data-action="save"` и т.п., чтобы bridge знал, что отправлять.
- Этот же файл (в сокращённом текстовом виде — список классов + примеры) кладётся в
  **системный промпт** агента, чтобы он генерировал разметку строго в рамках DS.
- Библиотека ассетов: обои, иконки приложений, glyph-набор.

### 3.5 Agent Bridge (backend, Node)
Локальный сервер (например, Fastify/Express + ws). Модули:
- **Intent Router** — принимает интенты от shell:
  `launch`, `event`, `close`, `menu-action`.
- **Prompt Builder** — собирает промпт: системный (правила VibeOs + DS-контракт) +
  контекст приложения (что за app) + событие/стейт формы.
- **Session Store** — на каждое окно своя Claude-сессия (приложение помнит свой
  стейт: текст в Notepad, корзину в Finder и т.п.).
- **Cache** — кэш по (appId + событие-хэш) для типовых экранов, чтобы не
  регенерировать и экономить токены/латентность.
- **Claude Code Runner** — вызывает агента (см. 3.6), нормализует вывод (вырезает
  всё кроме HTML-фрагмента), стримит в shell.

### 3.6 Интеграция с Claude Code
Два способа (рекомендуемый — SDK):
- **Claude Agent SDK** (`@anthropic-ai/claude-agent-sdk`, TS) — нативно из Node,
  стриминг токенов, управление сессиями (`resume`), системный промпт, ограничение
  инструментов. **Рекомендуется.**
- **CLI headless** как фолбэк: `claude -p "<prompt>" --output-format stream-json
  --session-id <uuid>` / `--resume <uuid>`. Парсим stdout.
- Для генерации UI агенту инструменты (file/bash) НЕ нужны — отключаем, оставляем
  чистую текст-генерацию → быстрее и безопаснее. (Позже Terminal-app сможет
  использовать инструменты осознанно.)

### 3.7 Output contract
Агент обязан вернуть **только HTML-фрагмент** (тело приложения), без markdown-обёрток
и пояснений. Backend дополнительно санитайзит: вырезает ```html```-заборы, проверяет,
что используются только разрешённые классы/теги, оборачивает в iframe-шаблон.
При генерации нового приложения агент также возвращает **метаданные** (короткое имя,
glyph/иконку, категорию) — они сохраняются в App Cache (3.10) для Spotlight/Launchpad.

### 3.8 Spotlight (лаунчер)
Лаунчер произвольных приложений.
- ⚠️ **⌘+Space нельзя использовать** — его перехватывает macOS (системный Spotlight)
  до браузера, веб-страница это событие не получает и не может отменить.
- Горячая клавиша: **⌘K** (стандарт для веб-командных палитр). Дополнительно —
  иконка-поиск 🔍 в menu bar и/или в доке, открывающая тот же оверлей кликом.
  (Опц. альтернативы, которые доходят до браузера: ⌘/ , Ctrl+Space.)
- Оверлей строки поиска открывается по центру экрана.
- Пользователь вводит **любое** название/описание ("Notepad", "Pomodoro timer",
  "JSON formatter"). Enter → `launch`-интент с этим именем → агент генерит UI.
- **Автоподстановка из кэша:** по мере ввода shell фильтрует App Cache (3.10) и
  показывает подходящие уже сгенерированные приложения первыми (мгновенный запуск
  без обращения к агенту), плюс варианты встроенных приложений. Если совпадений нет —
  предлагает "Generate '<запрос>'" (новая генерация).
- Ранжирование: точное совпадение → префикс → fuzzy; кэшированные выше «сгенерировать
  заново».

### 3.9 Launchpad
Полноэкранная сетка всех приложений (как в macOS).
- Источник — App Registry: встроенные + все кэшированные сгенерированные приложения.
- Клик по иконке = `launch` (для кэшированных — мгновенно из кэша).
- Поиск/фильтр сверху, удаление кэшированного приложения (forget), drag в dock
  (закрепить) — на этапе polish.

### 3.10 App Cache (приложения как сущности первого класса)
Сгенерированные приложения **сохраняются и переиспользуются**, а не живут только в
рамках одного окна.
- **Что храним:** `appId` (slug из имени), отображаемое имя, glyph/иконка, категория,
  первый отрендеренный HTML (стартовый экран), привязанный `sessionId` (опц.),
  таймстемпы (создано/последний запуск), счётчик запусков.
- **Где:** на backend (persist на диск, напр. JSON/SQLite), зеркало в shell для
  быстрого автокомплита Spotlight.
- **Как используется:**
  - Spotlight автоподстановка и Launchpad читают список из кэша.
  - Повторный запуск кэшированного приложения отдаёт стартовый HTML **мгновенно**
    (без вызова агента); агент подключается только на интерактив/обновления.
  - Кэш ключуется по нормализованному имени запроса (+ опц. хэш брифа), чтобы
    "notepad" и "Notepad" были одним приложением.
- Связь с §6 (Cache): App Cache = кэш приложений (первый экран + метаданные); в §6
  также кэш повторяющихся (appId+event-хэш) экранов взаимодействия.

## 4. Протоколы коммуникации

### 4.1 iframe → shell (postMessage)
```jsonc
{
  "type": "vibe-event",
  "windowId": "win-42",
  "event": {
    "action": "save",          // из data-action
    "element": "button#save",
    "value": null,
    "formState": { "title": "Заметка", "body": "..." } // собранные поля
  }
}
```
shell → iframe (команды): `{type:'vibe-cmd', cmd:'set-loading', target:'#save'}`.

### 4.2 shell → backend (WebSocket, JSON)
- `→ launch`   `{type:'launch', appId:'notepad', windowId}`
- `→ event`    `{type:'event', windowId, sessionId, event:{...}}`
- `→ close`    `{type:'close', windowId, sessionId}`
- `← render`   `{type:'render', windowId, html:'...', done:false}` (стрим чанками)
- `← status`   `{type:'status', windowId, state:'thinking|streaming|ready|error'}`

### 4.3 Поток «запуск приложения»
1. Клик по иконке Notepad в dock → shell создаёт окно (skeleton) + `launch`-интент.
2. Backend стартует сессию, шлёт системный промпт + "Сгенерируй стартовый UI
   приложения Notepad".
3. Стрим HTML → shell прогрессивно обновляет `srcdoc` (или показывает по `done`).
4. Окно `ready`.

### 4.4 Поток «взаимодействие»
1. Клик по кнопке внутри iframe → bridge.js → postMessage в shell.
2. Shell → backend `event` (с sessionId окна).
3. Backend resume-ит сессию: "Пользователь нажал Save. Состояние формы: …. Обнови
   UI." → новый HTML.
4. Shell перерисовывает iframe (full re-render в MVP; targeted DOM-patch — позже).

## 4bis. Примеры сценариев (целевое поведение)

Эти примеры задают планку и выводят важные принципы дизайна.

1. **«finder app but connected to pentagon server»** (⌘+Space).
   Открывается Finder, где папки/файлы — те, что *могли бы* лежать на сервере
   Пентагона (агент галлюцинирует контент под бриф). → **Free-text бриф целиком
   передаётся в промпт и определяет не только тип приложения, но и его контент и
   характер.** "finder" и "finder but pentagon" — это *разные* приложения (разный
   ключ кэша / запись в Launchpad).

2. **«terminal, but passive aggressive style»** (⌘+Space).
   Терминал отвечает пассивно-агрессивными фразами на любые команды и иногда даже
   "выполняет" их. → **Личность/поведение приложения — часть брифа.** Агент держит
   характер на протяжении сессии (см. §5), реагируя на ввод как на события.

3. **Safari → wikipedia.org → сгенерированная вики** с рабочими ссылками и поиском.
   → **Браузер — это мета-приложение.** Ввод URL — это интент `navigate`; агент
   генерит "страницу сайта". Клик по ссылке / submit поиска — события `navigate`/
   `search`, по которым агент рендерит следующую страницу. Внутренняя навигация =
   тот же цикл "событие → промпт → HTML", что и везде.

4. **Каждое приложение кладёт свою главную страницу в кэш и "прорастает" в
   Launchpad.** → Первый отрендеренный экран + метаданные сохраняются в App Cache
   (3.10); приложение появляется в Launchpad и в автоподстановке Spotlight; повторный
   запуск — мгновенный из кэша.

**Следствия для реализации:**
- **Prompt Builder** передаёт полный текст брифа дословно (личность, контекст,
  ограничения), а не только нормализованное имя.
- **Идентичность приложения для кэша** = нормализованный бриф (а не просто "finder").
  Отображаемое имя/иконку агент возвращает в метаданных.
- **Браузер** добавляет интенты `navigate(url)` / переходы по ссылкам — расширение
  протокола событий (§4.2), но механика та же.
- Браузерные "страницы" тоже можно кэшировать (url → HTML) для мгновенного возврата.

## 5. Управление состоянием и сессиями
- **1 окно = 1 Claude-сессия.** Контекст приложения живёт в диалоге сессии — это
  и есть "память приложения".
- Shell хранит зеркало стейта окон (для восстановления layout, повторного рендера).
- Закрытие окна → завершение/архивирование сессии.
- (Опц.) Персист сессий на диск → "ОС помнит" приложения между перезагрузками.

## 6. Кэширование и стоимость
- **App Cache** (3.10): стартовый экран + метаданные каждого сгенерированного
  приложения → мгновенный повторный запуск из Spotlight/Launchpad/dock.
- Кэш стартовых экранов приложений (детерминированный первый рендер).
- Хэш (appId + нормализованное событие) → готовый HTML для повторов.
- Стрим токенов, чтобы UX был живым, пока генерится.
- Отключённые инструменты у агента → короче/дешевле ответы.

## 7. Безопасность и модель интерактива

**Песочница — это граница безопасности** (а не санитайзер). Поэтому JS приложения
разрешён: он мощный, но изолированный.
- iframe `sandbox="allow-scripts"` **без** `allow-same-origin` (opaque origin):
  нет доступа к родителю, cookie, localStorage.
- CSP документа: `default-src 'none'` (никакой сети — ни fetch/XHR/WS), `img-src
  data: blob:`, `style-src 'unsafe-inline'`, `script-src 'unsafe-inline'
  'unsafe-eval'`. Итог: JS приложения может менять **только свой DOM**, ничего не
  слить и не загрузить.
- **Гибридный интерактив** (см. ds-prompt): локальный JS (`<script>`/`onclick`) для
  мгновенных детерминированных действий (калькулятор считает, табы, фильтры, игры,
  терминал с зашитой логикой) + `data-action` → агент только когда нужен новый
  галлюцинированный контент/характер/навигация.
- Санитайзер (`sanitize-html`) теперь нормализует структуру и allowlist
  тегов/атрибутов (включая `<script>`, обработчики `on*`, svg/canvas) — **но
  безопасность держится на песочнице+CSP**, не на нём. Сеть запрещена на всех
  уровнях (нет `href`/внешних `src`, только `data:`).
- Backend слушает только localhost; агент запускается изолированно
  (`settingSources: []`, `strictMcpConfig`, без инструментов).

## 8. Технологический стек (предложение)
| Слой            | Выбор                                   | Почему |
|-----------------|-----------------------------------------|--------|
| Shell UI        | TypeScript + Vite (vanilla или Lit)     | контроль, лёгкость, быстрый dev |
| Window Manager  | свой модуль на TS                       | полный контроль над drag/resize/z |
| Backend         | Node + Fastify + `ws`                   | простой локальный bridge |
| Agent           | `@anthropic-ai/claude-agent-sdk`        | стриминг, сессии; фолбэк — CLI |
| Стили окон/DS   | чистый CSS + CSS-переменные             | без зависимостей, легко в промпт |
| Санитайзер      | DOMPurify (на бэке через jsdom)         | безопасный AI-HTML |

> Альтернатива для shell: React. Минус — больше веса; плюс — проще сложный UI.
> Для MVP рекомендую vanilla TS, т.к. вся "тяжесть" в iframe'ах.
