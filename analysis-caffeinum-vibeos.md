# Анализ caffeinum/vibeOS — что перенять, а что сделать иначе

Источник: https://github.com/caffeinum/vibeOS (TypeScript, ~20★, последнее
обновление 2025-08).

> TL;DR: Это **не та же концепция**, что наша. У них приложения **захардкожены**
> как React-компоненты, а Claude Code работает как **чат-ассистент с root-доступом**,
> который реально выполняет команды (terminal, MCP, browser automation). У нас же
> приложения **генерируются на лету** (галлюцинированный софт). Поэтому брать стоит
> их *эстетику и интеграцию с Claude SDK*, а *архитектуру окон и модель приложений*
> делать иначе.

---

## 1. Что это на самом деле (реальная архитектура)

- **Стек:** Next.js 15 + React 19 + TypeScript + Tailwind v4 + tRPC + framer-motion
  + Radix UI + `@anthropic-ai/claude-code` SDK. Docker/Compose, Bun.
- **«Приложения» — это статичные React-компоненты**, написанные вручную:
  `notes-app.tsx`, `terminal.tsx`, `browser.tsx`, `crypto-tracker.tsx`,
  `file-browser.tsx`, `gcp-vm-list.tsx` и т.д. **Никакой генерации UI на лету нет.**
- **Рабочий стол (`src/app/page.tsx`)** — один большой компонент, где видимость окон
  управляется кучей булевых флагов (`showClaude`, `showTerminal`, `showBrowser`…) и
  «режимами» рабочего стола (`default / cluttered / focused / welcome`).
  Полноценного оконного менеджера (drag/resize/z-index/focus) фактически нет —
  «cluttered»-режим просто раскидывает окна по случайным координатам.
- **Роль Claude:** `src/app/api/chat/route.ts` — это чат-ассистент. Он получает
  сообщения (текст + картинки), стримит ответ через AI SDK и работает с
  `permissionMode: "bypassPermissions"`, имея доступ к shell и MCP-серверам.
- **Реальные действия:** через tRPC-роутеры (`terminal`, `files`, `gcp`, `crypto`,
  `browser-use`, `kernel`, `dedalus`) — например, `terminal.execute` тупо вызывает
  `child_process.exec(command)` на хосте.
- В репо есть `experiments/` (browser-use через CDP, dedalus) — явный
  hackathon-проект с scope creep.

**Главное различие с нами:** у них Claude = «умный ассистент поверх готовой ОС».
У нас Claude = «движок, который рисует сами приложения». Это две разные системы,
и большинство их архитектурных решений к нашей цели не подходят напрямую.

## 2. ✅ Что стоит перенять

### 2.1 Интеграция с Claude Code SDK (самое ценное)
`src/app/api/chat/route.ts` — почти готовый референс того, как мы будем дёргать агента:
- Использование `query({ prompt, options })` из SDK со **стримингом** токенов.
- **`appendSystemPrompt`** для впрыска правил VibeOs (у нас туда пойдёт Design System
  + output contract).
- **Сессии:** `options.continue = true` / `options.resume = sessionId` для
  продолжения диалога — ровно то, что нам нужно для «1 окно = 1 сессия».
- **Мультимодальность:** поддержка картинок во входе (image blocks) — пригодится,
  если захотим «покажи скриншот → собери похожий UI».
- ⚠️ Они импортируют пакет `@anthropic-ai/claude-code`; мы планировали
  `@anthropic-ai/claude-agent-sdk` (переименованный наследник). Надо свериться,
  какой пакет актуален на момент старта, API почти идентичен (`query`/`Options`).

### 2.2 macOS-эстетика и ассеты
- **`macos-icons.tsx`** — самописные SVG-иконки приложений (Finder, Safari,
  Terminal…). Отличная отправная точка для нашего dock'а.
- **`macos-dock.tsx`** — компонент дока с увеличением иконок.
- **Liquid/Glass-эффекты** (`ui/glass-effect.tsx`, `ui/liquid-glass.tsx`) — красивый
  «жидкое стекло» в стиле свежей macOS; можно перенести идею в наш CSS Design System.
- **framer-motion** для анимаций окон/меню — приятный UX (open/close, dropdown).
- Меню-бар с Apple-логотипом, часами, статус-иконками — готовый визуальный паттерн.

### 2.3 Идея «реальных возможностей» (опционально, на будущее)
- Их tRPC-роутеры (terminal/files/gcp/crypto) показывают, как приложение может
  делать **настоящие действия**, а не только рисоваться. Для нас это идея для
  поздних этапов: Terminal-app, который реально что-то исполняет (но в песочнице!).
- **Streaming через AI SDK** (`createUIMessageStream`) — удобный паттерн прогрессивной
  отдачи; у нас аналог пойдёт через WebSocket-чанки.

### 2.4 Компонентная UI-библиотека
- `components/ui/*` (button, tabs, tooltip, slider…) на Radix — напоминание, что нам
  нужен свой набор переиспользуемых элементов (у нас — CSS Design System вместо
  Radix, т.к. классы надо скармливать в промпт).

## 3. ❌ Что делать иначе (их слабые места / несовпадение с целью)

### 3.1 Приложения должны генерироваться, а не быть захардкожены
Это фундамент. У них каждый app — рукописный React. У нас цель из статьи Hackster —
**галлюцинировать software на лету** (даже Notepad). Поэтому:
- Сохраняем нашу модель: содержимое окна = **AI-сгенерированный HTML в iframe**, а не
  импортируемый компонент.
- Нужен **Design System как контракт для агента** + **санитайзер вывода** — у них
  этого нет (и не нужно, т.к. они не генерируют). У нас это критично.

### 3.2 Настоящий Window Manager вместо булевых флагов
`page.tsx` управляет окнами через десяток `useState(false)` и «режимы». Это не
масштабируется и не даёт реальной «оконности».
- Делаем **единый стор окон** (`WindowState[]`: id, geometry, zIndex, state,
  sessionId) и **общий оконный движок**: drag за titlebar, resize за 8 хэндлов,
  фокус/z-index по клику, minimize/maximize/close. (Наш M1.)
- Любое окно создаётся одинаково, независимо от приложения — никакого
  per-app `showX`-флага.

### 3.3 Безопасность: убрать root-доступ из браузера
Самый большой красный флаг репо:
- `permissionMode: "bypassPermissions"` + `terminal.execute` → `child_process.exec`
  **любой команды** с фронта, без allowlist, на хосте. Фактически RCE по дизайну.
- У нас наоборот: агент для генерации UI **без инструментов** (file/bash отключены);
  iframe в `sandbox="allow-scripts"` без `allow-same-origin`; санитайзинг HTML;
  backend только на localhost. Если позже добавим «реальные действия» — строго через
  allowlist и подтверждения, не bypass.

### 3.4 Не тащить тяжёлый стек без нужды
- Next.js + tRPC + Docker + Radix + Tailwind v4 — это много для нашей задачи. Вся
  «тяжесть» у нас в iframe'ах, поэтому мы выбрали **vanilla TS + Vite (shell)** +
  лёгкий **Node-bridge**. Не повторяем их монолит `page.tsx`.
- tRPC даёт типобезопасность, но добавляет связность фронт↔бэк; нам достаточно
  простого WebSocket-протокола с JSON-схемой.

### 3.5 Сессии — по окну, а не одна глобальная
У них `sessionId = "chat-session"` — одна захардкоженная сессия на весь чат.
- У нас **1 окно = 1 Claude-сессия** (маппинг windowId↔sessionId), чтобы каждое
  приложение помнило свой стейт независимо.

### 3.6 Фокус и дисциплина scope
- Их `experiments/` (browser-use CDP, dedalus, GCP, crypto) — расфокус хакатона.
- Мы держим MVP узким: window manager + design system + генерация UI + события из
  iframe. «Реальные интеграции» — только после того, как ядро работает.

## 5. Конкретные действия для нашего плана

1. **M3 (Agent Bridge):** взять `chat/route.ts` как референс вызова `query()` —
   стриминг + `appendSystemPrompt` + `continue/resume`. Свериться, какой пакет
   актуален (`claude-agent-sdk` vs `claude-code`).
2. **M2 (Design System):** позаимствовать визуальные идеи liquid-glass и SVG-иконки
   из `macos-icons.tsx`/`glass-effect.tsx`, переписав под наш чистый CSS.
3. **M1 (Window Manager):** сознательно сделать то, чего у них нет — настоящий
   оконный движок со стором, а не флаги видимости.
4. **Безопасность (сквозная):** зафиксировать как анти-паттерн их `bypassPermissions`
   + `exec(any)`; у нас агент генерации без инструментов, iframe в sandbox,
   санитайзер HTML, backend на localhost.
5. **Сессии:** маппинг 1 окно ↔ 1 сессия (а не одна глобальная).
6. **Scope:** не тащить их experiments; реальные интеграции — после рабочего ядра.
